# Стайл-гайд документации

Руководство по стилю и форматированию документации в этом репозитории.

## Общие принципы

1. **Ясность важнее красоты** - главное, чтобы было понятно
2. **Краткость сестра таланта** - короткие параграфы, простые слова
3. **Примеры обязательны** - показывайте, не только рассказывайте
4. **Единообразие** - следуйте установленным паттернам

## Структура документа

### Обязательные элементы

```markdown
# Заголовок документа (H1, один на страницу)

Краткое введение (1-2 параграфа) - что это за документ и зачем он нужен.

## Основной раздел 1 (H2)

Контент раздела в коротких параграфах (2-4 предложения).

### Подраздел (H3)

Детализация подраздела.

## См. также

- [Связанный документ 1](../path/to/doc.md)
- [Связанный документ 2](./another-doc.md)
```

### Рекомендуемые элементы

- Таблица содержания (для длинных документов)
- Примеры кода
- Диаграммы (где применимо)
- Admonitions (примечания, советы, предупреждения)
- Чеклисты (для пошаговых инструкций)

## Заголовки

### Правила

- **H1 (`#`)** - только один на страницу (название документа)
- **H2 (`##`)** - основные разделы
- **H3 (`###`)** - подразделы
- **H4-H6** - использовать только если действительно нужно

### Форматирование

- Все заголовки с заглавной буквы
- Без точки в конце
- Короткие и описательные

```markdown
✅ ## Установка зависимостей
❌ ## установка зависимостей.
❌ ## Как установить зависимости для проекта
```

## Текст

### Параграфы

- 2-4 предложения максимум
- Одна идея на параграф
- Пустая строка между параграфами

```markdown
✅ 
Это первый параграф. Он содержит одну мысль.

Это второй параграф. Он содержит другую мысль.

❌
Это очень длинный параграф который содержит множество мыслей и идей и его сложно читать потому что он не разбит на части и продолжается очень долго.
```

### Голос

- **Активный залог**: "Создайте файл" вместо "Файл должен быть создан"
- **Второе лицо**: "вы можете" вместо "пользователь может"
- **Настоящее время**: "команда выполняет" вместо "команда будет выполнять"

```markdown
✅ Установите пакет командой:
❌ Пакет может быть установлен следующей командой:
```

### Терминология

Используйте единую терминологию:

| Используйте | Вместо |
|-------------|--------|
| нажмите | кликните, жмите |
| запустите | запустить, выполнить |
| выберите | выбрать |
| установите | установить |

## Списки

### Нумерованные списки

Используйте для **последовательных действий**:

```markdown
1. Откройте терминал
2. Выполните команду
3. Проверьте результат
```

### Маркированные списки

Используйте для **перечислений** (порядок не важен):

```markdown
- Поддерживает Python
- Работает на Linux, macOS, Windows
- Легко настраивается
```

### Правила

- Максимум 7-8 пунктов в списке
- Для больших списков разбивайте на подкатегории
- Единообразное форматирование (все начинаются с глагола или все с существительного)

```markdown
✅
- Создает файл
- Обновляет конфигурацию
- Проверяет статус

❌
- Создает файл
- Конфигурация обновлена
- Проверить статус
```

## Admonitions

Используйте для выделения важной информации:

```markdown
!!! note "Примечание"
    Важная информация, которую нужно знать.

!!! tip "Совет"
    Полезный совет для оптимизации работы.

!!! warning "Внимание"
    Предупреждение о возможных проблемах.

!!! danger "Опасно"
    Критическое предупреждение о потере данных или ошибках.
```

### Когда использовать

- **note** - дополнительная информация
- **tip** - полезные советы и хаки
- **warning** - потенциальные проблемы
- **danger** - критические предупреждения

## Таблицы

### Простые таблицы

```markdown
| Колонка 1 | Колонка 2 | Колонка 3 |
|-----------|-----------|-----------|
| Значение  | Значение  | Значение  |
```

### Выравнивание

```markdown
| Влево     | По центру | Вправо    |
|:----------|:---------:|----------:|
| Текст     | Текст     | Текст     |
```

### Правила

- Используйте для сравнения данных
- Не используйте для длинного текста
- Максимум 5-6 колонок

## Форматирование

### Жирный и курсив

```markdown
**Жирный** - для важных терминов при первом упоминании
*Курсив* - для выделения, цитат
```

### Что НЕ делать

```markdown
❌ Не используйте CAPS для выделения
❌ Не используйте !!!множество!!! восклицательных знаков
❌ Не переусердствуйте с **жирным** текстом
```

## Примеры (полный документ)

### Хороший пример

```markdown
# Работа с API

Руководство по использованию REST API нашего сервиса.

## Аутентификация

Для доступа к API необходим токен. Получите его в личном кабинете.

### Получение токена

1. Войдите в личный кабинет
2. Перейдите в Settings → API
3. Нажмите "Generate Token"

```bash
# Использование токена
curl -H "Authorization: Bearer YOUR_TOKEN" \
  https://api.example.com/users
```

!!! tip "Совет"
    Храните токен в переменных окружения, не коммитьте в код.

## Endpoints

### GET /users

Получает список пользователей.

**Параметры:**

| Параметр | Тип    | Описание           |
|----------|--------|--------------------|
| page     | int    | Номер страницы     |
| limit    | int    | Элементов на странице |

**Пример:**

```python
import requests

response = requests.get(
    'https://api.example.com/users',
    headers={'Authorization': f'Bearer {token}'},
    params={'page': 1, 'limit': 10}
)
print(response.json())
```

## См. также

- [Аутентификация](authentication.md)
- [Примеры кода](../examples/)
```
